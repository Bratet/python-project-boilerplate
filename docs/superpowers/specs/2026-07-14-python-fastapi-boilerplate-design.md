# Python (FastAPI) Project Boilerplate — Design

**Date:** 2026-07-14
**Status:** Approved (pending spec review)

## Purpose

A ready-to-clone boilerplate that provides a clean, dockerized FastAPI runtime with
`uv`-managed dependencies. The goal: starting a new project should be "just write the
logic in `app/`" — the runtime environment, dependency management, containerization, and
quality tooling are already handled.

## Goals

- Clean, reproducible runtime via `uv` and a locked dependency set (`uv.lock`).
- Best-practice multi-stage Dockerfile with two build targets: `dev` and `prod`.
- Two first-class development paths:
  - **Container path (primary):** `docker compose` runs the dev container with
    hot-reload; `make` targets wrap it.
  - **Local path:** `uv sync` + `uv run …` directly on the host — no Docker required.
- Pre-wired quality tooling: Ruff (lint + format), Mypy (type checking), Pytest,
  pre-commit hooks.

## Non-Goals

- Not a sandbox for running the AI coding agent. The existing `make claude` target is a
  convenience only; the container is the app runtime, not an agent jail.
- No database, auth, CI pipeline, or deployment manifests in this iteration (can be added
  per-project later).

## Decisions (defaults)

| Decision | Value |
|---|---|
| App type | FastAPI web service |
| Environments | `dev` + `prod` (multi-stage targets in one Dockerfile) |
| Python version | 3.13 (via `ARG PYTHON_VERSION=3.13` and `.python-version`) |
| App port | 8000 |
| Package name | `app` |
| Dev workflow | docker compose + Makefile, plus local `uv` |
| Quality tooling | Ruff, Mypy, Pytest, pre-commit (all four) |

## File Layout

```
├── app/
│   ├── __init__.py
│   └── main.py            # FastAPI app + GET /health
├── tests/
│   └── test_main.py       # sample test hitting /health
├── Dockerfile             # multi-stage: base → builder → prod / dev
├── docker-compose.yml     # dev service: source-mounted, reload, port 8000
├── .dockerignore
├── pyproject.toml         # deps + dev group + ruff/mypy/pytest config
├── uv.lock                # generated by `uv lock`
├── .python-version        # 3.13
├── .pre-commit-config.yaml
├── .env.example           # copied to .env
├── Makefile               # keeps `claude`; adds compose + uv targets
└── README.md
```

## Component Design

### Dockerfile (multi-stage, uv best practices)

Single file with `# syntax=docker/dockerfile:1` and `ARG PYTHON_VERSION=3.13`.

- **`base`** — `python:${PYTHON_VERSION}-slim`. Copies the pinned `uv` binary from
  `ghcr.io/astral-sh/uv`. Sets `UV_COMPILE_BYTECODE=1`, `UV_LINK_MODE=copy`,
  `UV_PYTHON_DOWNLOADS=0`. `WORKDIR /app`.
- **`builder`** (from `base`) — installs dependencies in two cache-friendly layers:
  1. `uv sync --locked --no-install-project --no-dev` with
     `--mount=type=cache,target=/root/.cache/uv` and bind mounts for `uv.lock` +
     `pyproject.toml`. This layer's cache is invalidated only when the lockfile or
     `pyproject.toml` change, not on source edits.
  2. `COPY . /app` then `uv sync --locked --no-dev --no-editable` — installs the project
     itself (non-editable) into `/app/.venv`.
- **`prod`** (from fresh `python:${PYTHON_VERSION}-slim`) — creates a **non-root** system
  user, copies only `/app/.venv` from `builder` (the project is already installed into
  it, so no separate source copy is needed), puts `/app/.venv/bin` on `PATH`, `EXPOSE
  8000`, `USER app`, and runs
  `uvicorn app.main:app --host 0.0.0.0 --port 8000`.
- **`dev`** (from `base`) — `uv sync --locked --no-install-project` (includes the dev
  dependency group). Puts `/app/.venv/bin` on `PATH`, `EXPOSE 8000`, and runs
  `uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload`. Source is provided by the
  compose bind-mount at runtime, so edits hot-reload. The `app` package is importable
  because the working directory is `/app`.

### docker-compose.yml (dev)

One `app` service:
- `build: { context: ., target: dev, args: { PYTHON_VERSION: "${PYTHON_VERSION:-3.13}" } }`
- `ports: ["8000:8000"]`
- `env_file: .env`
- `volumes:`
  - `.:/app` — live source mount for hot-reload
  - `venv:/app/.venv` — **named volume** so the container's built environment is not
    shadowed by the host bind-mount (Docker seeds a fresh named volume from the image
    contents on first run)
- top-level `volumes: { venv: {} }`

After changing dependencies, `make sync` re-runs `uv sync` inside the container to update
the `venv` volume (or `docker compose down -v` to reset it).

### Makefile

Extends the existing `claude` target. Grouped:

- **Container (compose):** `up`, `down`, `logs`, `shell` (exec bash into the dev
  container), `sync` (re-run `uv sync` in the container), `build-prod`
  (`docker build --target prod`), `run-prod`.
- **Local / quality (via `uv run`, work on host *or* inside container):** `install`
  (`uv sync`), `lint` (`ruff check .`), `format` (`ruff format .`), `typecheck`
  (`mypy .`), `test` (`pytest`), `check` (lint + typecheck + test, CI-style), `hooks`
  (`pre-commit install`).
- **Existing:** `claude` (`claude --dangerously-skip-permissions`).

### pyproject.toml

- `[project]` — name `app`, `requires-python = ">=3.13"`, dependencies `fastapi`,
  `uvicorn[standard]`.
- `[dependency-groups] dev` — `ruff`, `mypy`, `pytest`, `httpx` (for the FastAPI
  `TestClient`), `pre-commit`.
- `[tool.ruff]` — lint + format config.
- `[tool.mypy]` — strict-ish (e.g. `strict = true` with pragmatic relaxations if needed).
- `[tool.pytest.ini_options]` — test paths, etc.

### app/main.py

Minimal FastAPI app exposing `GET /health` returning `{"status": "ok"}`. This is the
single place a user starts writing their own logic.

### tests/test_main.py

Uses FastAPI `TestClient` to assert `GET /health` returns 200 and the expected body.
Serves as the sample that `make test` runs green out of the box.

### Supporting files

- `.dockerignore` — excludes `.git`, `.venv`, `__pycache__`, caches, `.env`, etc., to keep
  the build context small.
- `.env.example` — documented placeholders; copied to `.env` (gitignored) for compose.
- `.python-version` — `3.13`, so local `uv` selects the right interpreter.
- `.pre-commit-config.yaml` — runs Ruff (lint + format) and Mypy on commit.
- `README.md` — quickstart for both paths (compose and local uv), plus common `make`
  targets.

## Data Flow

- **Request flow (runtime):** client → port 8000 → uvicorn → FastAPI (`app.main:app`) →
  `GET /health` handler → JSON response.
- **Dev build/run flow:** `make up` → `docker compose build (target=dev)` → deps installed
  into `venv` volume → container runs uvicorn `--reload` with host source bind-mounted →
  edits on host trigger reload.
- **Prod build flow:** `make build-prod` → multi-stage build → `builder` installs project +
  deps into `.venv` → `prod` copies only `.venv`, runs as non-root.

## Error Handling & Edge Cases

- **Named venv volume staleness:** if dependencies change, the `venv` volume must be
  refreshed via `make sync` or `docker compose down -v`. Documented in README and Makefile
  help.
- **Host/container venv collision:** the named `venv` volume prevents a host `.venv` from
  leaking into the container and vice versa.
- **Lockfile drift:** all `uv sync` calls use `--locked` (dev/prod builds fail if the
  lockfile is out of date rather than silently resolving).

## Testing Strategy

- `tests/test_main.py` covers the `/health` endpoint via `TestClient`, runnable both
  locally (`make test`) and in the container.
- `make check` runs Ruff, Mypy, and Pytest — the same gate a CI job would run.
- Manual verification: `make up` then `curl localhost:8000/health` returns
  `{"status":"ok"}`; `make build-prod && make run-prod` serves the same.

## Open Questions

None. Defaults (Python 3.13, port 8000, package `app`) are chosen and easily changed
later per project.
